## 1. 客户端开发环境（Electron、Tauri）

当前项目选择的是 Tauri，不是 Electron。

原因主要有这几个：

- Tauri 更适合做一个“本地壳 + 本地 runtime 管理器”的产品，体积更小，系统集成更直接。
- Kadaclaw 需要本地文件系统能力、进程拉起能力、配置目录管理能力、安装 OpenClaw 到私有目录的能力，这些都很适合走 Tauri command + Rust。
- 如果用 Electron，同样也能做，但安装包体积、运行时占用和分发成本通常会更高。

当前客户端技术栈是：

- Tauri 2
- React 18
- Ant Design
- Zustand
- Rspack
- Rust

整体架构是：

1. React 负责桌面 UI
2. Tauri 负责前端和本地系统能力的桥接
3. Rust 负责调用本地 openclaw CLI、管理配置、安装 runtime、同步 skills
4. OpenClaw 作为被客户端托管的本地 runtime 运行

## 2. 如何和 OpenClaw 进行 WebSocket 通信

需要先说明一件事：

当前项目里，聊天窗口不是直接自己手写 WebSocket 协议去和 OpenClaw 通信的。当前实现走的是更稳的一层封装：

1. 前端调用 Tauri command
2. Rust 层执行本地 openclaw agent --session-id ... --message ...
3. OpenClaw 返回结果
4. Rust 把结果回传给前端

也就是说，当前聊天主链路是 CLI 驱动，不是前端直连 WebSocket。

但是 OpenClaw runtime 本身是有 gateway 的，本地通常会暴露一个 gateway 地址，例如：

- ws://127.0.0.1:18789

当前项目里和 OpenClaw 的几类通信方式分别是：

- Runtime 存活检测：HTTP 探测本地 endpoint
- Dashboard 嵌入：通过 openclaw dashboard --no-open 解析控制台地址
- 聊天调用：通过 openclaw agent CLI 发消息
- 技能状态同步：通过 openclaw skills list --json

如果后续要改成“真正的 WebSocket 直连模式”，可以演进成下面这种结构：

1. 客户端启动时拉起本地 OpenClaw gateway
2. 前端或 Rust 层建立到本地 gateway 的 WebSocket 连接
3. 会话消息、状态流、事件回推都走 WS
4. CLI 只保留安装、升级、配置、技能管理等管理类能力

当前项目还没有走到这一步，原因是：
CLI 方式更稳定、兼容性更高，也更适合作为第一版产品落地。

## 3. 如何打包 OpenClaw 和客户端

当前方案不是“把 OpenClaw 二进制静态塞进仓库”，而是“客户端内置安装器”。

### 当前做法

Kadaclaw 自己打包成 Tauri 桌面应用；OpenClaw 不在构建时直接嵌进去，而是在首次运行或用户点击按钮时，由客户端自动安装到应用私有目录。

安装过程大致是：

1. 用户启动 Kadaclaw
2. 点击“一键安装内置 OpenClaw”
3. Rust 层执行 OpenClaw 官方安装脚本
4. 将 OpenClaw 安装到 Kadaclaw 私有目录
5. Kadaclaw 自动把运行配置切到这个内置 runtime

当前私有安装目录类似：

- ~/Library/Application Support/com.kadaclaw.app/openclaw-runtime

里面包含：

- bin/openclaw
- config/openclaw.json
- state/
- skills/

### 这种方案的优点

- 用户不需要先手工全局安装 OpenClaw
- OpenClaw 和 Kadaclaw 生命周期绑定，更像完整产品
- 不污染系统级目录
- 后续升级 runtime 更容易做应用内升级

### 当前方案的限制

- 首次安装依赖联网
- 还不是离线捆绑交付
- 如果以后要做企业内网分发，可能还要补“离线 runtime 包”方案

### 后续可升级的打包方案

后面可以演进成两种分发方式：

1. 在线安装模式
   客户端包只带安装器，首次运行拉取 OpenClaw 最新版
2. 离线捆绑模式
   客户端安装包直接包含一个指定版本的 OpenClaw runtime，安装时直接展开到私有目录

## 4. 如何在启动客户端时也启动 OpenClaw，是否需要 OpenClaw Control UI

### 客户端启动时如何启动 OpenClaw

当前项目已经在做这件事。

逻辑大致是：

1. Kadaclaw 启动
2. 调用 ensure_openclaw_runtime
3. 检查本地 runtime 是否存在、是否可达
4. 如果没运行，则尝试自动启动
5. 如果已经在线，则不重复启动

当前也已经补了“防重复启动”逻辑，避免每次进客户端都重复拉起 OpenClaw。

### 是否需要 OpenClaw Control UI

这取决于产品定位。

#### 方案 A：保留 Control UI

适合早期产品阶段。

优点：

- 能快速复用 OpenClaw 已有控制台能力
- 减少你自己重做 runtime 管理界面的成本
- 对调试和开发有帮助

缺点：

- 用户体验上会有“嵌了另一个产品页面”的感觉
- 一体化程度不如原生客户端界面

#### 方案 B：不依赖 Control UI

适合后期产品化阶段。

做法是：

- Kadaclaw 自己提供聊天、模型配置、runtime 状态、技能管理界面
- OpenClaw 只作为底层 runtime 和 gateway
- Control UI 仅作为开发/调试工具，不在正式用户路径中暴露

### 当前建议

当前项目可以保留 Control UI 作为辅助面板，但主路径应该逐步转向：

- 聊天窗口由 Kadaclaw 自己实现
- 设置页由 Kadaclaw 自己实现
- 技能市场由 Kadaclaw 自己实现
- OpenClaw 只提供 runtime 能力

也就是说，长期看不应该依赖 OpenClaw Control UI 作为主界面。

## 5. 客户端如何安装 Skills

当前项目已经实现了基础版本地安装链路。

### 当前安装方式

技能市场里的“安装”动作不是只改前端状态，而是会真正写到内置 OpenClaw 私有 skills 目录里。

当前落盘结构大致是：

- openclaw-runtime/skills/<skill-id>/skill.json
- openclaw-runtime/skills/<skill-id>/SKILL.md

同时，Kadaclaw 会自动把这个目录注册进 OpenClaw runtime 配置里的：

- skills.load.extraDirs

这样 OpenClaw 就可以去扫描并识别这些本地技能。

### 当前技能安装流程

1. 用户在技能市场点击安装
2. 前端调用 Tauri command
3. Rust 层在私有 skills 目录下创建技能目录
4. 写入 skill.json
5. 写入 SKILL.md
6. 更新 OpenClaw runtime 配置中的 skills.load.extraDirs
7. 调用 openclaw skills list --json 同步识别状态
8. 前端显示“已安装 / 已识别 / 已就绪”

### 当前实现状态

已经做到：

- 技能目录真实落盘
- OpenClaw 能识别这些技能
- 前端能展示技能识别状态

还没完全做完的是：

- 生成的 skill 内容目前还是脚手架级别
- 还没做到完整的远程市场下载和版本化升级
- 还没做技能包签名、校验、来源可信度控制

### 后续建议

后续可以把 skills 安装演进为三层：

1. 本地安装层
   安装到私有 skills 目录
2. 市场分发层
   从远程市场拉取技能包、元数据、版本信息
3. 更新与安全层
   支持升级、卸载、回滚、签名校验、来源验证
